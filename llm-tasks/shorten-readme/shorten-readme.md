# Shorten the README

`README.md` is ~178 lines and front-loads operational depth (proxy lifecycle, sandbox
internals, Nix consent flow) that a first-time reader does not need. Shorten it to a
practical first-encounter doc, and relocate the still-accurate detail that is not already
covered elsewhere into one new reference file. This is a **documentation-only** change:
no C#, XAML, or test code is touched.

## Current state (researched)

`README.md` carries these sections (by heading): `## Install`, `## Run (source checkout)`,
`## Model Backend` (with `### Provider keys`), `## Sandbox`, `## What It Does` (a 12-bullet
feature list), `## Development Rails`, `## Tests`. It opens with a one-paragraph intro and
`![Visual Relay main window](docs/images/visual-relay-main.png)`.

Most of the bulk is already duplicated by existing files:

- `docs/DESIGN.md` — architecture, the 11-stage Relay mapping, GUI views, execution model,
  safety. Covers the same ground as the README's `## What It Does` bullet list.
- `AGENTS.md` — contributor dev tooling, the control API, sample/run-task tools, and the
  "Build & checks" rails (`./visual-relay build|test|launch|check`, the commit-msg hook,
  the 300-line guard, `[AvaloniaFact]`).
- `TROUBLESHOOTING.md` — test hangs, the `[AvaloniaFact]` rule, and a table of backend XDG
  paths (`$XDG_DATA_HOME/visual-relay/backend-venv`, `…/scratch/litellm.pid`, `…/litellm.log`).
- `.env.example` — the full provider key set, resolution precedence
  (process env > user `~/.config/visual-relay/.env` > repo `.env`), and both key locations.
  This already duplicates the README's `### Provider keys` subsection.
- The `visual-relay` launcher script self-documents the Nix consent-gated install flow in its
  header comment, so the README's multi-paragraph Nix consent explanation is redundant.

The genuinely unique, worth-keeping operational detail not covered by the above is: the
LiteLLM proxy lifecycle (auto-start, `VisualRelay.Backend start|stop|status`, key-aware
generated config at `…/scratch/litellm-config.generated.yaml`) and the sandbox internals
(nono always-on, the self-healing `vr-guard` profile at
`$XDG_CONFIG_HOME/visual-relay/vr-guard.json`, and `sandboxExtraAllowPaths` in
`.relay/config.json`).
