# Port `tools/backend/backend.sh` to a published `VisualRelay.Backend` tool

`tools/backend/backend.sh` (381 lines) is the litellm local-proxy lifecycle manager
(`start`/`stop`/`status`): it resolves per-machine state under XDG, starts the proxy, polls
readiness on `http://127.0.0.1:4000`, and stops it with SIGTERM→SIGKILL. It ships in the
release tarball and is invoked from `visual-relay launch`. Move its logic into a tested,
published C# tool and retire the script.

This design is decided — implement exactly this, no alternatives:

- New **`tools/VisualRelay.Backend`** (Exe, `net10.0`), **published self-contained** in the
  release like `VisualRelay.Init`/`VisualRelay.GenBackendConfig`, with subcommands
  `start` / `stop` / `status`.
- It **reuses the app's existing C# backend lifecycle** rather than re-porting bash: the app
  already starts the backend and probes readiness in C# (see
  `llm-tasks/DONE-autostart-model-backend-on-launch.md`, `StartBackendAsyncTests`,
  `BackendReadinessProbeTests`). Extract/share that lifecycle so the GUI autostart and this
  CLI tool run the **same** code.
- Per-machine state stays under XDG (`$XDG_*_HOME/visual-relay/…`), **never the repo tree**
  (host+VM share the folder). Python the proxy spawns must not write bytecode — set
  `PYTHONDONTWRITEBYTECODE=1` (the nono `__pycache__` stall, `visual-relay:21-29`).
- `backend.sh` is **retired**: the release ships the published `VisualRelay.Backend`, and
  `visual-relay launch` (now `VisualRelay.Cli`, task 13) calls it.

## Current state (researched)

- **`backend.sh`** is the lifecycle manager (start idempotent + readiness poll; stop
  SIGTERM→SIGKILL; status), with state under `$XDG_DATA_HOME/visual-relay/` and a static
  template at `tools/backend/litellm-config.yaml`. It is invoked from `visual-relay:488`
  (`"$SCRIPT_DIR/tools/backend/backend.sh" start || …`) and shipped in the release layout.
- **The app already does this in C#:** backend autostart on launch + a readiness probe exist
  (`StartBackendAsyncTests`, `BackendReadinessProbeTests`, `StartBackendAsyncTests`,
  `BackendStatusIndicatorTests`; `llm-tasks/DONE-autostart-model-backend-on-launch.md`,
  `DONE-backend-request-hard-timeout.md`). The config is generated by
  `VisualRelay.GenBackendConfig` (`BackendConfigGeneratorTests*`). **Verify the exact seam
  names** and lift the shared lifecycle into `Core` so both the GUI and this tool use it.
- **`backend.sh` has characterization tests** that must be re-pointed or replaced:
  `Installer5BackendShTests(.Runtime/.Bytecode/.GenConfigTimeout)` drive the script directly
  (incl. asserting `PYTHONDONTWRITEBYTECODE` behavior). Preserve those guarantees against the
  C# tool.
- **Tool/publish conventions:** mirror `tools/VisualRelay.Init` (csproj + top-level
  `Program.cs`); register in `VisualRelay.slnx`; add the publish step to
  `.github/workflows/release.yml` alongside the existing `dotnet publish … Init …` /
  `… GenBackendConfig …` steps, and update the release-layout copy that currently bundles
  `tools/backend/`.
